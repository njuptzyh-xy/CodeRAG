# CodeRAG 项目改造计划书

## 1. 文档目的

本文档用于统一 CodeRAG 后续改造方向，明确：

- 项目为什么要改
- 一期到底要做到什么
- 先改什么，后改什么
- 每个阶段的交付物是什么
- 哪些旧能力保留，哪些旧设计要退出主链路

本文档在以下已有材料基础上整理形成：

- `CODEX_CONFIGURATION_GUIDE.md`
- `GENERAL_CODE_ANALYSIS_REFACTOR_PLAN.md`
- 当前仓库现状


## 2. 项目背景

当前 CodeRAG 最初面向“红队知识库”场景建设，已经具备以下基础能力：

- 项目或文件上传
- 文件扫描与代码切块
- LLM 分析
- Milvus 向量检索
- Neo4j 图谱存储
- 基础问答链路

但当前系统仍存在明显的历史包袱：

- 数据模型带有较强安全领域语义
- 主流程默认围绕红队知识和 ATT&CK 语义组织
- 通用代码分析与领域增强逻辑没有分层
- 上传、解析、存储、检索职责混杂

这使得项目虽然“看起来像一个代码 RAG”，但实际上还不是一个真正面向任意代码项目的通用分析平台。


## 3. 改造总目标

将 CodeRAG 从“红队知识库构建系统”改造为“通用代码项目解析与 RAG 平台”。

目标架构应满足以下要求：

- 能解析任意代码项目及其相关文件
- 能构建项目级、文件级、代码块级知识索引
- 能支持围绕项目内容的检索、问答和关系追踪
- 安全领域能力不再作为主链路前提，而是作为可选插件接入

可用一句话概括目标：

`CodeRAG = 通用代码分析底座 + 通用检索问答能力 + 可插拔领域增强能力`


## 4. 一期建设范围

一期建议聚焦“通用代码项目解析与问答”的最小可用闭环，不追求一步到位。

### 4.1 一期目标

- 支持导入本地项目目录、压缩包形式的项目
- 支持扫描代码、文档、配置、依赖描述文件
- 支持项目级摘要、关键文件识别、模块识别
- 支持代码块与文档块的结构化入库
- 支持项目作用域内的检索与问答
- 支持基础关系图谱构建

### 4.2 一期重点支持对象

- Python
- Java
- Go
- JavaScript / TypeScript

### 4.3 一期非目标

- 不承诺支持所有语言和所有构建体系
- 不直接兼容全部旧版红队图谱 schema
- 不追求完整静态分析器能力
- 不在一期完成全部安全插件化能力


## 5. 核心设计原则

### 5.1 通用能力优先

新主链路必须先在没有 ATT&CK、没有红队知识节点的情况下独立运行。

### 5.2 领域增强后置

安全分析、ATT&CK 映射、安全标签等能力保留，但只作为插件层接入。

### 5.3 渐进迁移

不建议直接全量替换旧逻辑。优先采用：

- 新增通用模型
- 新增通用流程
- 逐步迁移调用方
- 最后下线旧主链路

### 5.4 模块职责清晰

上传、解析、存储、检索、问答、插件增强应分层实现，避免继续堆积在 `service/` 中。

### 5.5 先闭环，再扩展

先完成“项目导入 -> 解析 -> 入库 -> 检索 -> 问答”的通用闭环，再扩展更多高级能力。


## 6. 当前系统主要问题

### 6.1 数据模型不通用

当前核心对象中仍存在明显领域耦合，不适合作为通用平台主模型。

### 6.2 文件处理范围偏窄

当前更偏向代码文件处理，对 README、配置、依赖清单、接口定义、SQL 等支持不足。

### 6.3 检索链路偏向旧知识库问答

现有检索更像“知识节点召回”，不够适合项目理解类问题，例如：

- 项目入口在哪
- 某模块依赖什么
- 某个配置影响哪些逻辑
- 某个接口的调用链路是什么

### 6.4 服务层职责混杂

当前部分 `service/` 同时承担编排、LLM 调用、图谱写入、向量写入、文件处理，后续维护成本高。

### 6.5 存储设计仍偏旧场景

Milvus 和 Neo4j 的字段、关系、命名仍然更贴近旧红队知识组织方式，而不是项目知识组织方式。


## 7. 目标能力模型

改造后的平台建议围绕以下五类核心能力建设：

### 7.1 项目接入

- 支持项目目录导入
- 支持压缩包导入
- 后续预留 Git 仓库接入能力

### 7.2 项目解析

- 文件扫描
- 文件分类
- 语言识别
- 代码切块
- 文档切块
- 配置提取
- 依赖提取
- API 与外部服务信息提取

### 7.3 知识建模

- 项目
- 目录
- 文件
- 代码块
- 文档块
- 配置项
- 模块
- 函数 / 类
- 依赖项
- 外部服务

### 7.4 存储与索引

- Milvus 负责向量检索
- Neo4j 负责关系表示
- 元数据负责过滤、聚合、范围约束

### 7.5 检索与问答

- 项目理解型问答
- 文件定位型问答
- 依赖关系型问答
- 配置影响型问答
- 调用链路型问答


## 8. 目标目录与模块分层

建议逐步收敛到以下结构：

```text
CodeRAG/
  routes/
  service/
    api/
  core/
    ingestion/
    code_analysis/
    retrieval/
  clients/
    llm/
    graph/
    vector/
  repositories/
  models/
    domain/
    dto/
  plugins/
    security/
  scripts/
  docs/
```

目录职责说明：

- `routes/`：HTTP 路由入口
- `service/api/`：接口级业务编排
- `core/ingestion/`：项目导入、解包、扫描
- `core/code_analysis/`：项目解析、切块、结构抽取
- `core/retrieval/`：检索规划、召回、重排、答案组装
- `clients/`：外部组件访问封装
- `repositories/`：存储抽象层
- `models/`：领域模型与接口 DTO
- `plugins/`：领域增强插件


## 9. 新的数据模型建议

建议以“通用项目知识模型”为主线，逐步替代旧安全领域模型。

### 9.1 核心实体

- `Project`
- `Directory`
- `File`
- `CodeChunk`
- `DocChunk`
- `ConfigItem`
- `Module`
- `Class`
- `Function`
- `Dependency`
- `APIEndpoint`
- `ExternalService`
- `AnalysisResult`

### 9.2 关键元数据

每个知识单元建议至少带以下字段：

- `project_id`
- `project_name`
- `file_path`
- `language`
- `content_type`
- `symbol_name`
- `symbol_kind`
- `start_line`
- `end_line`
- `summary`
- `tags`
- `hash`

### 9.3 核心关系

- `PROJECT_CONTAINS_DIRECTORY`
- `DIRECTORY_CONTAINS_FILE`
- `FILE_CONTAINS_CHUNK`
- `FILE_DEFINES_CLASS`
- `FILE_DEFINES_FUNCTION`
- `MODULE_DEPENDS_ON_MODULE`
- `FUNCTION_CALLS_FUNCTION`
- `FILE_IMPORTS_FILE`
- `FILE_USES_CONFIG`
- `PROJECT_DEPENDS_ON_PACKAGE`
- `PROJECT_EXPOSES_API`
- `PROJECT_CONNECTS_TO_SERVICE`

### 9.4 安全领域关系

安全相关关系作为插件层扩展，例如：

- `CODE_IMPLIES_ATTACK_TECHNIQUE`
- `PROJECT_RELATES_TO_TACTIC`
- `FILE_HAS_SECURITY_CAPABILITY`

这些关系不再进入通用主链路的必填设计。


## 10. 一期主链路设计

一期建议先形成以下统一流程：

### 10.1 导入阶段

- 接收项目目录或压缩包
- 解压或准备工作目录
- 创建分析任务
- 记录项目快照信息

### 10.2 解析阶段

- 扫描文件
- 识别文件类型
- 提取代码文件、文档文件、配置文件
- 对代码做 AST 或规则切块
- 对文档做段落切块
- 对配置做键值或分段提取

### 10.3 分析阶段

- 生成项目摘要
- 识别关键文件
- 识别入口文件
- 识别模块职责
- 提取依赖与服务信息
- 提取配置项与接口信息

### 10.4 入库阶段

- 写入项目、文件、chunk 元数据
- 写入向量索引
- 写入图谱关系

### 10.5 检索问答阶段

- 识别问题类型
- 生成检索计划
- 在项目作用域内召回
- 融合向量、关键词、关系结果
- 返回答案与证据


## 11. 分阶段实施计划

## Phase 0：基线梳理与冻结

### 目标

在正式改造前，先稳定当前系统，避免无序修改。

### 工作项

- 梳理现有主链路
- 标记红队耦合点
- 梳理当前可用能力与必须保留能力
- 输出旧系统能力基线

### 交付物

- 旧主链路说明
- 红队耦合点清单
- 现有能力保留清单


## Phase 1：模型去红队化

### 目标

建立通用项目知识模型，替换旧主结果结构。

### 工作项

- 新建通用实体模型
- 新建通用分析结果 schema
- 将主流程输出从安全语义切换为项目语义
- 将安全 prompt 从主流程剥离
- 停止继续扩散 `MitreAttack*` 命名

### 交付物

- 新领域模型
- 新分析结果结构
- 迁移映射说明


## Phase 2：重构项目解析流水线

### 目标

让系统具备对任意代码项目做基础理解的能力。

### 工作项

- 重构文件扫描器
- 增加文件分类能力
- 增加语言识别与项目类型识别
- 增加入口文件识别
- 增加关键模块识别
- 增加配置项提取
- 增加依赖项提取
- 增加 API/外部服务信息提取
- 调整 chunk 策略

### 交付物

- 新 `analyze_project` 主流程
- 通用项目分析结果 JSON schema
- 示例项目分析产物


## Phase 3：重构存储层

### 目标

让存储模型围绕项目知识组织，而不是围绕旧知识库挂载。

### 工作项

- 重做 Neo4j schema
- 重做 Milvus collection 字段
- 建立统一存储抽象接口
- 设计项目、文件、chunk、关系的写入链路
- 明确新旧 schema 并行或迁移方案

### 交付物

- 新图谱 schema
- 新向量库 schema
- 数据迁移策略


## Phase 4：重构检索与问答

### 目标

让系统能围绕“项目理解”进行检索和回答。

### 工作项

- 重新定义问题类型
- 设计多路召回策略
- 增加项目作用域过滤
- 支持文件、模块、函数粒度返回
- 增加图谱辅助召回
- 重做 rerank 输入结构
- 统一答案与证据返回结构

### 交付物

- 新查询能力设计
- 新检索策略定义
- 新响应格式


## Phase 5：安全能力插件化

### 目标

在通用平台稳定后，再将原有安全能力以插件形式接回。

### 工作项

- 抽离 ATT&CK 映射逻辑
- 抽离安全领域 prompt
- 抽离安全标签与规则
- 定义插件接口
- 支持按项目或任务启用插件

### 交付物

- 安全插件接口
- ATT&CK 映射插件
- 安全增强说明文档

## 13. 里程碑建议

### M1：完成通用模型定义

判断标准：

- 新主结果结构可表达项目、文件、chunk、关系
- 新功能不再引入新的安全领域核心命名

### M2：完成通用解析闭环

判断标准：

- 能对至少一种真实项目完成导入、分析、结构化输出

### M3：完成通用入库闭环

判断标准：

- 项目、文件、chunk、基础关系可写入存储

### M4：完成项目问答闭环

判断标准：

- 能回答入口、模块、依赖、配置、接口定位类问题

### M5：完成安全插件回接

判断标准：

- 安全能力不影响通用主链路，且可按需启用


## 14. 风险与应对

### 14.1 旧数据兼容风险

风险：

- 新旧 schema 不一致，已有数据难以直接复用

应对：

- 新旧 schema 并行一段时间
- 提供迁移脚本
- 分批切流

### 14.2 需求边界失控

风险：

- “支持任意项目”过于宽泛，导致改造范围持续膨胀

应对：

- 一期明确限制语言与项目类型
- 优先保证主流场景闭环

### 14.3 LLM 稳定性不足

风险：

- 项目类型变复杂后，模型输出波动变大

应对：

- 提高结构化输出约束
- 增加规则提取与回退逻辑
- 关键环节尽量减少纯生成式依赖

### 14.4 图谱维护成本过高

风险：

- 如果图谱关系设计过重，维护成本可能高于收益

应对：

- 先只保留对检索和关系问答真正有价值的关系
- 不在一期追求完备调用图


## 15. 推荐协作方式

结合 `CODEX_CONFIGURATION_GUIDE.md`，后续建议采用以下协作方式：

- 默认单主控 agent，按阶段推进
- 大任务必须分轮次执行
- 先分析影响面，再做改造
- 多 agent 只在任务边界清晰时使用
- 子任务按目录和职责拆分，避免重叠写入

推荐执行顺序如下：

1. 先做 Phase 0 和 Phase 1
2. 再做 Phase 2，形成通用解析闭环
3. 再做 Phase 3 和 Phase 4，形成通用检索问答闭环
4. 最后做 Phase 5，把安全能力插件化接回


## 16. 下一步建议

如果按可执行性排序，建议下一步直接开展以下三项工作：

1. 输出“旧系统耦合点清单”和“必须保留能力清单”
2. 设计新的通用领域模型与分析结果 schema
3. 选定一个真实项目样本，作为新解析链路的验证对象

完成这三项后，再进入代码层面的第一轮重构，会更稳。
