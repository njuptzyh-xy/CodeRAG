# CodeRAG 通用代码分析平台改造路线图

## 1. 背景

当前项目最初面向“红队知识库”场景建设，核心流程默认依赖：

- MITRE ATT&CK 相关图谱实体
- 红队知识预置节点
- 面向安全语义的提示词与分析结果
- 面向安全知识检索的 Neo4j + Milvus 数据组织方式

这导致系统虽然已经具备“代码上传、代码切块、LLM 分析、向量检索、图谱查询”的基础能力，但整体仍然与“红队知识”强耦合，难以直接扩展为“分析任意代码项目”的平台。

本次改造的目标不是简单替换字段名，而是将系统重构为：

`通用代码项目分析平台 = 通用代码分析底座 + 可插拔领域增强能力`


## 2. 改造目标

### 2.1 总体目标

将当前项目从“红队知识库构建系统”改造为“支持任意代码项目分析、建库、检索和问答的通用平台”。

### 2.2 一期建议目标

建议先收敛到以下范围：

- 支持上传并分析主流代码仓库
- 支持项目级摘要、模块识别、关键文件识别
- 支持文件/代码块级结构化入库
- 支持项目内关系建图
- 支持基于项目内容的检索与问答
- 保留后续接入安全领域增强能力的扩展点

### 2.3 非目标

一期不建议直接承诺：

- 支持所有语言和所有项目类型
- 一步到位实现完备的静态分析引擎
- 保持与旧版红队知识图谱完全兼容
- 在不改 schema 的前提下完成通用化


## 3. 核心设计原则

### 3.1 领域解耦

把“通用代码分析能力”和“红队/安全领域知识映射能力”彻底拆开。

- 通用层只关注项目本身事实
- 领域层只负责解释、标注和扩展

### 3.2 先项目内知识，后外部知识

新系统必须先能在没有 ATT&CK、没有红队预置节点的情况下独立完成项目分析和问答。

### 3.3 数据模型通用化

所有核心实体与关系都应从“红队术语”迁移到“软件工程术语”。

### 3.4 可插拔增强

后续安全分析、架构分析、依赖风险分析等能力都应作为插件或策略层接入，而不是写死在主流程中。


## 4. 现状问题

### 4.1 数据模型强耦合

当前存在大量强领域实体命名，例如：

- `MitreAttackCodeSoftware`
- `MitreAttackTechnique`
- `MitreAttackTactic`
- `MitreAttackArticleDocument`

这些命名会直接限制系统的分析边界和输出表达能力。

### 4.2 分析目标偏安全语义

当前分析链路更关注：

- 项目对应哪些战术
- 文件对应哪些技术
- 代码如何挂靠到 ATT&CK 节点

而通用代码分析更需要回答：

- 这个项目是做什么的
- 项目入口在哪里
- 核心模块有哪些
- 模块依赖和调用关系如何
- 配置、服务、数据库、接口在哪

### 4.3 检索链路不适用于通用项目问答

当前检索链路偏向“安全知识召回”，而不是“项目理解型问答”。这会影响：

- chunk 组织方式
- 元数据设计
- 返回结构
- rerank 目标

### 4.4 模块职责混杂

当前 `service/` 中大量逻辑同时承担：

- 文件处理
- LLM 调用
- 图谱写入
- 向量写入
- 业务编排

后续一旦扩展通用分析能力，维护成本会快速上升。


## 5. 目标架构

建议将系统拆为 5 层。

### 5.1 接入层

负责：

- 上传项目
- 创建分析任务
- 查询分析结果
- 发起问答请求

对应目录建议：

- `routes/`
- `service/api/`

### 5.2 分析编排层

负责：

- 管理项目分析流程
- 串联扫描、切块、摘要、结构抽取、关系抽取
- 控制异步任务与重试策略

对应目录建议：

- `app/analysis/`
- `app/pipeline/`

### 5.3 通用代码分析层

负责：

- 文件扫描
- 语言识别
- AST/规则切块
- 入口识别
- 关键文件识别
- 模块摘要
- 调用/依赖/配置关系抽取

对应目录建议：

- `core/code_analysis/`

### 5.4 存储抽象层

负责：

- 图数据库写入与查询
- 向量库写入与检索
- 元数据存储

对应目录建议：

- `repositories/`
- `clients/`

### 5.5 领域增强层

负责：

- ATT&CK 映射
- 安全标签提取
- 领域知识挂载
- 特定行业规则分析

对应目录建议：

- `plugins/security/`
- `plugins/domain/`


## 6. 新的数据模型建议

### 6.1 核心实体

建议逐步迁移到如下通用实体：

- `Project`
- `Directory`
- `File`
- `CodeChunk`
- `Class`
- `Function`
- `Module`
- `ConfigItem`
- `Dependency`
- `APIEndpoint`
- `DatabaseTable`
- `ExternalService`
- `Document`

### 6.2 核心关系

- `PROJECT_CONTAINS_DIRECTORY`
- `DIRECTORY_CONTAINS_FILE`
- `FILE_CONTAINS_CHUNK`
- `FILE_DEFINES_CLASS`
- `FILE_DEFINES_FUNCTION`
- `MODULE_DEPENDS_ON_MODULE`
- `FUNCTION_CALLS_FUNCTION`
- `FILE_IMPORTS_FILE`
- `FILE_USES_CONFIG`
- `PROJECT_DEPENDS_ON_PACKAGE`
- `PROJECT_EXPOSES_API`
- `PROJECT_CONNECTS_TO_SERVICE`

### 6.3 领域扩展关系

安全领域可作为附加关系存在，例如：

- `CODE_IMPLIES_ATTACK_TECHNIQUE`
- `PROJECT_RELATES_TO_TACTIC`
- `FILE_HAS_SECURITY_CAPABILITY`

这类关系不应再成为主链路的必需前提。


## 7. 一期建议输出能力

一期建议把输出结果聚焦为以下几类：

- 项目摘要
- 关键文件列表
- 模块职责摘要
- 项目入口与启动方式
- 配置项清单
- 对外接口清单
- 依赖组件清单
- 外部服务连接信息
- 文件级和代码块级语义索引
- 项目内部关系图


## 8. 分阶段实施计划

## Phase 0：冻结旧逻辑并建立迁移基线

### 目标

在改造前先稳定当前系统，避免“边改边失控”。

### 工作项

- 梳理现有主链路：
  - `upload_code`
  - `upload_file`
  - `query`
  - `neo4j_route`
- 标记红队耦合点：
  - schema
  - prompt
  - service
  - helper
  - analyzer 输出
- 确定哪些旧能力必须保留
- 输出一版旧系统能力基线文档

### 交付物

- 旧系统链路图
- 红队耦合点清单
- 保留能力清单


## Phase 1：命名与模型去红队化

### 目标

先把“概念层”拆出来，建立新通用模型，不立即迁移全部旧数据。

### 工作项

- 新建通用领域模型
- 将分析结果对象从 `software/tactic/technique` 迁移为通用结构
- 重命名 `red_kbs_analyzer`
  - 可先保留旧目录，新增新包承接通用逻辑
- 将安全专属 prompt 从主流程抽离
- 将 Neo4j 节点定义从 ATT&CK 命名改为通用命名

### 注意点

- 不建议直接全局替换旧类名
- 建议新增通用模型层，再逐步迁移调用方

### 交付物

- 新实体模型定义
- 新关系模型定义
- 通用分析结果 schema


## Phase 2：重构通用代码分析流水线

### 目标

让系统在不依赖红队知识的前提下，完成任意项目的基础理解。

### 工作项

- 重构文件扫描器
- 增加语言识别与项目类型识别
- 增加入口文件识别
- 增加关键模块识别
- 增加配置项提取
- 增加依赖项提取
- 增加 API/服务连接信息提取
- 调整 chunk 策略，使其更适合项目问答

### 建议输出结构

- 项目级摘要
- 文件级摘要
- 代码块级摘要
- 结构化元数据

### 交付物

- 新的 `analyze_project` 主流程
- 通用项目分析结果 JSON schema


## Phase 3：重构存储层

### 目标

让存储结构围绕“项目知识”而不是“红队知识挂载”组织。

### 工作项

- 重做 Neo4j schema
- 重做 Milvus collection 字段
- 统一存储抽象接口
- 将 `ESHelper` 之类历史兼容命名替换为真实角色命名
- 设计项目、文件、chunk、关系的写入流程

### 建议 Milvus 索引单元

- 项目摘要向量
- 文件摘要向量
- 代码块内容向量
- 配置项描述向量

### 建议 Neo4j 侧重点

- 层级关系
- 调用关系
- 依赖关系
- 配置与外部服务关系

### 交付物

- 新图谱 schema
- 新向量库 schema
- 数据迁移策略


## Phase 4：重构检索与问答能力

### 目标

让问答系统围绕“项目理解”工作，而不是围绕 ATT&CK 节点回填工作。

### 工作项

- 重新定义问答类型
- 重新设计检索策略
- 增加项目作用域过滤
- 增加文件/模块/函数粒度返回
- 增加图谱关系辅助召回
- 重做 rerank 输入结构

### 推荐支持的问题类型

- 项目入口在哪
- 某个功能实现在哪些文件
- 某个模块依赖哪些组件
- 某个配置影响哪些逻辑
- 某个接口的处理链路是什么
- 某个类/函数被谁调用

### 交付物

- 新 `/query` 能力设计
- 新检索策略定义
- 新响应格式


## Phase 5：将安全能力改造成插件

### 目标

在通用平台稳定后，再把原有红队/安全能力以插件形式接回。

### 工作项

- 抽离 ATT&CK 映射逻辑
- 抽离安全领域 prompt
- 抽离安全规则和标签体系
- 定义插件接口
- 让分析流程支持可选插件执行

### 交付物

- `security plugin` 插件接口
- ATT&CK 映射插件
- 安全标签增强插件

## 11. 风险与难点

### 11.1 旧数据兼容问题

如果线上已有大量 Neo4j/Milvus 数据，直接换 schema 会带来迁移成本。建议：

- 新旧 schema 并行一段时间
- 提供迁移脚本
- 分阶段切流

### 11.2 “任意项目”边界失控

如果不限制语言和项目类型，分析复杂度会迅速失控。建议一期只支持：

- Python
- Java
- Go
- JavaScript/TypeScript

### 11.3 图谱价值不清

如果只是做语义检索，图数据库可能会变成高维护负担。建议先明确：

- 是否真的需要调用链、依赖链、配置关系查询
- 如果需要，保留 Neo4j
- 如果不需要，可先以向量库 + 元数据为主

### 11.4 LLM 结果稳定性

原系统对 LLM 的依赖很重。通用项目分析后，输入项目类型更复杂，结果波动会更明显。需要增加：

- 结构化输出约束
- 回退策略
- 规则提取兜底

## 13. 建议的实施方式

建议采用“并行新建 + 渐进迁移”，不要在原主链路上直接大改。

推荐策略：

1. 新建通用模型与新分析链路
2. 保留旧红队链路继续可用
3. 逐步将上传、分析、检索切换到新链路
4. 最后再决定旧链路保留范围

这样做的好处是：

- 风险更低
- 更容易回滚
- 更容易对比新旧效果


## 14. 下一步建议

建议你下一步先做下面 3 件事：

1. 先确定一期支持的语言和项目类型范围
2. 先确定新的通用实体模型和分析结果 schema
3. 先确定 Neo4j 是否保留，以及保留到什么程度

如果这 3 件事没有先定下来，后续改造很容易陷入“所有地方都要改，但改完仍然没有明确目标”的状态。
